Welcome to the Odysseus AI Setup Hub

						

							The official Odysseus AI Setup Hub is designed as a comprehensive orchestration directory and installation reference for developers, engineers, and data privacy advocates. Odysseus AI functions purely as a self-hosted workspace cockpit interface that connects to your local machine resources; it is not a standalone AI model engine itself. Managing a local artificial intelligence pipeline requires precise configuration of backend runtimes, database layers, and front-end interface containers. Our Docker Guide and companion PewDiePie Install checklist walk you through initializing the local repository, managing local system dependencies, and resolving common environment variable issues. Deploying local AI allows complete data isolation, preventing telemetry leaks and ensuring sensitive files remain secure on your own local hardware setups.
						

					


					

						
Docker Container Environment Orchestration

						

							Using containers is the most reliable way to run the local workspace. This Docker Guide explains how to launch and configure the monorepo application using Docker Compose networks. The standard deployment architecture isolates individual services into separate containers: the web client Nginx server, the Express API routing backend, and the PocketBase database platform. A robust container network isolates dependencies, allowing direct integration with ChromaDB vector stores for RAG, SearXNG search modules for real-time web retrieval, and local Ollama model runners. Setting up Docker requires configuring the environment file (.env) with valid keys, binding container network ports securely, and mounting external Docker volumes to ensure that user settings, data collections, and auth tables persist across system reboots.

Windows Native Setup and CUDA GPU Acceleration

						

							For developers running on Windows 10 or 11, running natively outside of container virtual layers yields maximum execution speed by talking directly to graphics processing unit (GPU) drivers. Performing a native Windows installation requires downloading Git for Windows, installing Python 3.11+ (adding it to the system environment PATH variables), and installing the NVIDIA CUDA Toolkit version 12.1 or newer. This ensures that PyTorch and local model engines can tap into the tensor cores of NVIDIA RTX cards. Once the prerequisites are met, developers can run our PowerShell installer script to check hardware configurations, download workspace dependencies, and verify directory pathways automatically.

macOS Metal Framework Local Acceleration Guide

						

							Apple Silicon hardware (M1, M2, and M3 chips) features unified memory architectures that offer massive VRAM capacity for running large language models locally. Bypassing container networks on macOS is highly recommended to leverage the unified memory pool. Setting up Odysseus AI natively on a MacBook requires installing Xcode Command Line Tools, configuring the Homebrew package manager, and setting up a clean virtual Python environment. Using Apple's Metal Performance Shaders (MPS) allows the models to execute matrix operations directly on GPU cores rather than CPU threads. Our macOS guide outlines step-by-step instructions to compile local wheel files, verify Python pathings, and map unified memory boundaries.

Ollama Endpoint Mapping and Model Configuration

						

							Connecting the dashboard to a model runner like Ollama is the core step in achieving a functional workspace. When the workspace cockpit runs inside a Docker container, it cannot reach the host machine's loopback address (127.0.0.1) directly because containers use an isolated bridge network. To map the Ollama base URL, you must change the endpoint address from localhost to the host gateway ip (typically host.docker.internal:11434 on Windows and macOS, or the default bridge IP on Linux). Once mapped, Odysseus AI can pull available local GGUF models, query embeddings, and manage system prompts securely.

Troubleshooting Port Conflicts and Authentication Failures

						

							Most setup blockages stem from port conflicts or database migration issues. By default, Odysseus AI binds the web cockpit dashboard to port 7000. If port 7000 is occupied by another local service (such as Apple AirPlay Receiver on macOS), you must rebind the port in the docker-compose or environment file. Furthermore, the first login requires a secure SQLite database connection initialized via Better Auth migrations. If the dashboard displays login errors or fails to load workspace logs, our Triage Wizard and diagnostic checklists provide immediate solutions to rebuild database tables, verify network sockets, and inspect volume access parameters.

Frequently Asked Questions

						
Does Odysseus AI require an internet connection?

						

							No, when paired with local runners like Ollama, Llama.cpp, or LM Studio, Odysseus AI operates completely offline, ensuring that your data, prompts, and code never leave your machine.
						

						
What is the minimum GPU requirement for local LLMs?

						

							We recommend a minimum of 8GB of VRAM to run quantized 7B models (like Llama-3 or Mistral) comfortably. For larger models (like 14B or 32B), a minimum of 16GB or 24GB of VRAM is recommended. Use our VRAM calculator page to check your hardware profile.
						

						
How do I update the Odysseus AI container files?

						

							To update to the latest release, navigate to your local repository directory and execute a git pull command: git pull origin main. Then, rebuild the container network using: docker compose up -d --build.
